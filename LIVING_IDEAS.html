Code Similarity Engine

9 ideas
  • Code matching must be completely unobstructed
    ArchitecturePriority
    Do NOT let genealogy pre-filter which files get compared. Compare everything against everything. The most valuable discoveries are files that don't look related by name but share significant code. Independence from genealogy is a design constraint.
  • Code similarity explains why individually-passing files break together
    VisionPriority
    6 files each pass their own tests. App breaks. Code similarity finds the interface drift — a module in one file that needs to align with modules in other files doesn't match. This is the core use case. Not archaeology. Live diagnostics.
  • Code similarity should run all the time
    Product
    Like a type checker — you don't run it once. An analysis step can easily turn into an ongoing real-time scan. But dead apps in standby mode need to be leveled up to real-time. They can get there, but there's a progression.
  • Name overlap without code overlap is its own signal
    Technical
    Files with matching names but no matching code means someone rewrote from scratch. Did they lose important logic? Also: files with <1% code match across all other files tells a lot — either truly independent or disconnected from the system it should be part of.
  • Taxonomy of similarity patterns — index them all
    Architecture
    As long as we can index all similarity pattern types, and use Opus to read them and get full context for each file, that will be enough to understand the patterns. The taxonomy is the foundation.
  • Archaeology mode for dead apps
    ProductMarketing
    Users with lots of dead apps want to bring them back to life. The archaeological scan tells them what they have. Framing matters — how you present "here's your dead code" determines whether users feel empowered or overwhelmed.
  • Similarity taxonomy: 8 pattern types
    Technical
    Function clone, structural mirror, partial extraction, template variant, evolution pair, import fingerprint, dead copy, logic overlap. Each pattern type has different implications for what to do about it.
  • Two modes: full scan + incremental
    Architecture
    Full scan: compare everything against everything (batch, run once). Incremental: re-compare changed files against known relationships (real-time, on every edit via PostToolUse hook). Mode 1 first, Mode 2 makes it a product.
  • Code similarity is independent of chat history
    ProductArchitecture
    This engine reads source code, not conversations. A user who never processes a single session can still use it. Standalone capability that integrates with but doesn't depend on the rest of the pipeline. Dependency flows: similarity → genealogy → visualizations.

The Leveling System

3 ideas
  • Apps in standby mode can't jump to real-time
    VisionPriority
    You can't just real-time past apps that are in standby mode. They can get there, but they have to be leveled up. There's a progression from dead → understood → enhanced → monitored.
  • Analysis naturally evolves into real-time scans
    Product
    An analysis step can easily turn into an ongoing real-time scan. The transition from batch to continuous is natural — but it requires the foundational analysis to exist first.
  • Codebase maturity levels
    Product
    Level 0: Dead app on disk. Level 1: Archaeological scan (similarity + genealogy). Level 2: Chat history processed (semantic primitives, idea graph). Level 3: Hyperdocs embedded. Level 4: Real-time monitoring. Each level builds on the previous.

  • Multiple onboarding scenarios must be tested

    Product

  • The 10 commitments are the most valuable output

  • Semantic primitives are genuinely innovative

    MarketingVision

Pipeline & Architecture

7 ideas
  • Pipeline state: 285 sessions, 73% credibility verified
    Technical
    285 sessions, 6,135+ JSONs, 456 hyperdocs, 341 enhanced files, 18 genealogy families (5 signals). 261 fully complete. 24 partial. Phase 5 ground truth: 73% credibility (22,812 checks). Completeness scanner now tracks every field. 3 cron jobs syncing.
  • The transformation proof: frustration → constraint
    TechnicalVision
    Traced message 1108 ("ALWAYS WITH LLM!!!!!!!") through 6 phases. Entered as raw text with caps_ratio 1.0 and tier 1 (SKIP) — too short for standalone processing. But Phase 1 agents read session context and placed it in react-003. Phase 2 built it into GM-003 at 0.99 confidence. A 45-byte frustrated outburst, classified as noise by the filter, became the highest-confidence architectural constraint in the session. The pipeline's value is contextual synthesis, not message filtering.
  • Code similarity feeds three downstream systems
    Architecture
    Similarity results feed into: (1) genealogy as a grounding signal, (2) ground truth verifier as a consistency check, (3) hyperdocs as per-file warnings about interface drift. One engine, three consumers.
  • Build code similarity BEFORE visualizations
    ArchitecturePriority
    The dependency chain: code similarity script → enriched genealogy → trustworthy visualizations. Building visualizations first means visualizing incomplete data. The script is the missing station on the production line.
  • Schema variance: the distributed agent consistency problem
    TechnicalPriority
    Completeness scanner revealed that 261 sessions produce Phase 1 files, but only 30% have the expected 6-thread structure. Only 7% of semantic_primitives have tagged messages. Bulk agents (50-60 concurrent) used inconsistent output schemas. Fix: either schema validation at write time or normalization at read time.
  • Completeness scanner as data provenance tool
    Architecture
    Distinguishes POPULATED/ZERO/MISSING/FILE_MISSING at field level across all sessions. The data dictionary defines what should exist; the scanner checks what actually does. Together they close the "genuinely zero vs not collected" gap. Output: completeness_report.json (831KB).
  • Interface drift detection as product differentiator
    ProductVision
    "Your app doesn't work because these two files disagree about how data flows between them, and here's exactly where they diverge." Nobody else offers this. It's integration testing at the semantic level — not "do tests pass?" but "do these files agree?"

Feb 10, 2026 — Onboarding Test: Fresh Session Phase 0

Test subject: Session 02552cd9 (99 messages, 35 user / 64 assistant, 362 KB JSONL). Never previously processed.

What worked:
• deterministic_prep.py ran cleanly. Produced enriched_session.json (0.2 MB, 99 messages) in <2 seconds.
• prepare_agent_data.py ran cleanly. Produced all 9 Phase 0 files + batch dir + sanitized 2 files (profanity removal).
• completeness_scanner.py correctly identified the session as "missing phase_1, phase_2, phase_3, phase_5" — Phase 0 complete.
• app_level_manager.py still reads Level 5 VERIFIED (existing 261 sessions anchor the level). Session count: 285 → 285.
• Tier distribution: 82 SKIP (83%), 3 basic (3%), 10 standard (10%), 4 priority (4%). Zero frustration peaks, zero emergencies.
• Token usage parsed: 4.8M input, 5.3K output, 7.9K thinking chars.

What broke / issues found:
• Bug: Nested output directory. When HYPERDOCS_OUTPUT_DIR is set to the full path including session_XXXX, get_session_output_dir() appends another session_XXXX/ inside it. First attempt produced session_e7be321d/session_e7be321d/. Fix: set HYPERDOCS_OUTPUT_DIR to the parent sessions directory, not the target session dir. A new user would hit this immediately.
• No documentation on env vars. The script requires HYPERDOCS_SESSION_ID, HYPERDOCS_CHAT_HISTORY, and HYPERDOCS_OUTPUT_DIR but there's no README or help text showing the correct values. The relationship between these three vars is non-obvious.
• Code similarity not exercised. Session only mentioned 1 file (MEMORY.md), which isn't Python. code_similarity_incremental.py --file has nothing to scan. Short sessions with few code operations won't benefit from this layer.
• Content policy sanitization runs on ALL files including safe_* files that already stripped content. Harmless but wasteful on second pass.

What a real user would experience:
The $0 foundation layer works. A user with a JSONL file can run 2 commands and get a fully enriched session in seconds. The main friction is knowing which env vars to set and how OUTPUT_DIR interacts with get_session_output_dir(). The concierge system (which automates this setup) would eliminate this friction entirely.
