3 — FILE GENEALOGY

What It Does

AI doesn't version files — it rewrites them under new names. A user sees 40 files and feels overwhelmed. File genealogy shows them: "You actually have 12 concepts, and here's the latest version of each." It detects file identity across renames, rewrites, and duplications using three independent signals.

Three Detection Signals

  • Idea-graph lineage — Concept A (mentions file X) evolved into concept B (mentions file Y) via idea graph edges. Strongest signal.
  • Temporal succession — File X stops being modified, file Y starts at the same time (within 5 messages).
  • Name similarity — Files with overlapping stems. thread_extractor.py and six_thread_extractor.py share the core stem.

Implementation

  • phase_2_synthesis/file_genealogy.py — 450 lines
  • Inputs: thread_extractions.json (Phase 1) + idea_graph.json (Phase 2)
  • Output: file_genealogy.json (per session)
  • Algorithm: Union-find clustering with strict merge rules
  • Consumed by: Phase 3 collect_file_evidence.py and Phase 4a aggregate_dossiers.py

Missing Orchestration

run_pipeline.py Phase 2 runs idea_graph.py and synthesis.py — but does NOT call file_genealogy.py. It must be run manually after Phase 2. If it's not run, Phase 3 gets empty genealogy data.

4 — CODE SIMILARITY

What It Does

Compares all Python files against all others. No pre-filtering. No genealogy input. Completely independent of chat data. Reads source files on disk, extracts AST fingerprints (functions, classes, imports, constants), and classifies every pair into one of 10 relationship patterns.

Five Fingerprint Signals Per File

  • Function names — AST-extracted top-level functions
  • Class names + method signatures — AST classes and their methods
  • Import fingerprint — What modules are imported
  • Constants and string literals — ALL_CAPS names, strings >10 chars
  • Filename stem words — Split by underscore

Implementation

  • phase_0_prep/code_similarity.py — 520 lines Primary
  • phase_2_synthesis/code_similarity.py — 517 lines Duplicate
  • Input: Raw Python files on disk (independent of chat pipeline)
  • Output: code_similarity_index.json (7.9 MB, in ~/PERMANENT_HYPERDOCS/indexes/)
  • Consumed by: Phase 0 prepare_agent_data.py (NEW), Phase 4a aggregate_dossiers.py

The Duplicate Problem

Two near-identical implementations exist in phase_0_prep/ and phase_2_synthesis/. No comments indicate why or which is canonical. This creates maintenance burden and divergence risk.

5 — THE CIRCULAR DEPENDENCY

File Genealogy Cannot Be a Phase 1 Input

File genealogy requires thread_extractions.json (a Phase 1 output) to build file activity timelines. It also requires idea_graph.json (a Phase 2 output) for its strongest signal: idea-graph lineage. This creates a hard circular dependency.

SignalRequiresAvailable at Phase 0?Notes
Idea-graph lineageidea_graph.json (Phase 2)NoNeeds concept evolution edges. Only available after full synthesis.
Temporal successionFile mention timestampsYesMetadataExtractor already extracts file mentions per message.
Name similarityFilename listYesAll file mentions in Phase 0 metadata. Pure string comparison.

A "lite genealogy" running at Phase 0 using only temporal succession and name similarity would give Phase 1 agents partial file relationship data. This has not been implemented.

The Budget Bomb Problem

5 sessions (mentioning 275-824 files) would consume 50-87% of the 865K token budget with code similarity context alone, leaving almost no room for actual message data. Current implementation has no cap.

Worst case: session_f81f1051 (293 files, 4,284 matches) = ~752K tokens = 87% of budget.

Design decision needed — options:

  • Option 1: Signal score threshold — Only include matches above score X (e.g., >1.0)
  • Option 2: Match count cap — Max N matches per session (e.g., 500)
  • Option 3: Token budget cap — Limit context to X% of total budget (e.g., 10%)
  • Option 4: Truncate shared_functions — Remove function name lists for large contexts

The Dead Copy Paradox

The global index contains 15 dead copies (all pXX_enforcer.py vs root__pXX_enforcer.py pairs). But 0 dead copies survive session filtering — users interact with one copy, never both in the same session. The most actionable pattern for Phase 1 is actually function_clone (1,089 matches across sessions).

9 — CODE SIMILARITY: 10 PATTERN TYPES
PatternThresholdsWhat It MeansAction
dead_copy>90% textVirtually identical file. One is redundant.Delete one
evolution_pair60-90% textSame file diverged over time. Version drift.Reconcile
function_clone>50% funcs, <90% textCopy-paste with edits. Shared function names.Extract shared
partial_extraction>80% containmentOne file's functions are a subset of the other's.By design
template_variant>40% callable, <30% textSame structure, different content.By design
import_twin>70% imports, <30% funcsSame ecosystem, different purpose.Informational
name_only>30% name, <5% codeMisleading names. Filenames overlap but code doesn't.Informational
interface_mismatch3+ shared funcs, <30% textAPI drift. Same function names, diverged implementations.Fix drift
structural_sibling>15% callable, >30% importsSame general structure and ecosystem.Informational
shared_ecosystem>40% imports, >10% callablePart of the same import ecosystem.Informational

Scoring Formula

signal_score = func_overlap * 3 + callable_overlap * 2 + class_overlap * 2 + import_overlap * 1 + constant_overlap * 1 + name_overlap * 1

Text comparison (difflib) only runs on pairs where signal_score >= 0.1 or name_overlap > 0.3. This optimization means ~80% of the 57,970 pairs avoid the expensive text diff step.

10 — OUTPUT SCHEMAS
file_genealogy.json — Per-Session Schema
  • session_id string — Session UUID
  • file_families[] array — Grouped files representing the same concept
  •   .concept — Human-readable concept name
  •   .versions[] — Each version: file, status (current/superseded), active_msgs, events, evidence
  •   .latest — Filename of current version
  • standalone_files[] — Files with no detected relatives
  • links_detected — Counts by signal type: idea_graph, temporal, name_similarity, total
  • reduction — E.g., "60 files → 45 concepts"
code_similarity_index.json — Global Index Schema
  • total_files integer (341) | total_pairs_compared integer (57,970) | non_trivial_matches integer (14,176)
  • pattern_distribution — Counts per pattern type
  • matches[] — Every non-trivial pair: file_a, file_b, signals (11 fields), patterns[]
  • file_stats{} — Per-file: fingerprint, total_matches, pattern_summary, strongest_match, isolation_score
code_similarity_context.json — NEW Per-Session Filtered Context
  • session_id string — Session UUID
  • generated_at ISO timestamp — When context was generated
  • source_index — "code_similarity_index.json"
  • session_files_count integer — Files mentioned in session
  • matches_included integer — Number of filtered matches
  • matches[] — Filtered matches where both file_a and file_b are in session file set
  •   .file_a, .file_b — Filenames
  •   .signals — 11 fields: func_overlap, callable_overlap, class_overlap, import_overlap, constant_overlap, name_overlap, text_similarity, containment_a_in_b, containment_b_in_a, shared_functions[], signal_score
  •   .patterns[] — 0-4 of the 10 pattern types
  • file_stats{} — Per session-mentioned file: fingerprint, total_matches, pattern_summary, strongest_match, isolation_score
11 — WHAT'S STILL MISSING

1. No Token Budget Cap

20 sessions (7.4%) risk consuming >20% of the token budget with code similarity context. 5 sessions would consume >50%. Design decision needed.

2. File Genealogy Still Disconnected

Circular dependency unresolved. The "lite genealogy" idea (temporal succession + name similarity at Phase 0) remains unimplemented.

3. Duplicate code_similarity.py

phase_0_prep/code_similarity.py (520 lines) and phase_2_synthesis/code_similarity.py (517 lines) — near-identical. Which is canonical? No resolution yet.

4. file_genealogy.py Not in run_pipeline.py

Must be run manually after Phase 2. If not run, Phase 3 gets empty genealogy data and lineage sections are blank.

5. No Dead Copy Visibility for Phase 1

Session filtering requires both files to be mentioned. Since dead copies are never co-mentioned, Phase 1 agents will never see dead_copy alerts. Consider including dead copies where either file is mentioned.

6. No Signal Score Threshold

Low-signal matches (score <1) add noise without actionable insight. A minimum threshold would reduce token cost and improve signal-to-noise ratio.
