Finding 1 Multiple Writers to the Same Data File

Eleven data files in the pipeline are written by more than one script. When two scripts write to the same file, only one script's output survives at runtime. If both scripts run during a pipeline execution, the second writer silently overwrites the first writer's output. Whether this is a problem depends on the intent: some multi-writer patterns are by design, while others represent duplicate implementations that should be consolidated.

The eleven multi-writer data files fall into three categories, analyzed below.

Category A: Schema Normalizers (Intentional)

Two scripts in phase_0_prep/ — schema_normalizer.py and normalize_agent_output.py — both read and write the same nine JSON files. These nine files are the primary outputs of the Opus extraction agents in Phases 1 through 3:

This is a deliberate pipeline pattern. LLM agents produce JSON output with inconsistent schemas — the same semantic data filed under different key names, arrays where dicts are expected, missing required fields. The schema normalizers run after agent output is complete, read each file, apply canonical transformations (key renaming, structure flattening, field injection), and write the file back in place. They create atomic backups before writing, and they log every transformation to normalization_log.json.

The fact that there are two separate normalizer scripts doing the same job to the same files (schema_normalizer.py and normalize_agent_output.py) is itself a duplicate that was flagged and resolved earlier in this session — they were near-identical, with normalize_agent_output.py being the version with structured logging. The older post_agent_normalize.py was moved to obsolete.

Note: While the normalizer pattern is intentional, having two normalizer scripts that do the same thing (schema_normalizer.py and normalize_agent_output.py) is still redundant. They were identified as a duplicate pair earlier, but both were kept because they have slightly different import chains. Consolidating them into one canonical normalizer would simplify the pipeline.

All five scripts read the same five Phase 1 input files (thread_extractions.json, geological_notes.json, semantic_primitives.json, explorer_notes.json, and session_metadata.json), perform synthesis analysis, and write the same three output files. Despite having identical I/O signatures, the code is not identical. Diffs between the four Phase 2 scripts range from 234 lines (the closest pair: backfill_phase2.py vs batch_p2_generator.py) to 1,161 lines (the most divergent pair: backfill_phase2.py vs batch_phase2_processor.py). They represent four separate attempts at implementing Phase 2 synthesis, written at different times, that were never consolidated into a single canonical version.

Recommendation: Consolidate to a single canonical implementation. The strongest candidate is backfill_phase2.py because it uses the standard pipeline utilities (tools.json_io for file I/O and tools.log_config for structured logging), while batch_p2_generator.py and batch_phase2_processor.py use raw print() statements and inline json.load() calls. The batch orchestration pattern (iterating all sessions) should be handled by a separate runner script that calls the canonical implementation, not by duplicating the synthesis logic.

Finding 2 Phase Boundary Violations

Five scripts read data files that are produced by phases that run after them in the pipeline sequence. A script in Phase 0 cannot meaningfully run during Phase 0 if it requires output from Phase 1, Phase 2, or Phase 3. These scripts are misplaced in the directory structure — they live in phase_0_prep/ or phase_1_extraction/, but they functionally operate as post-processing utilities that should run after all extraction and synthesis phases are complete.

Why This Matters

The pipeline is designed to run in sequential phases: Phase 0 prepares raw data, Phase 1 extracts structure via LLM agents, Phase 2 synthesizes connections, Phase 3 writes dossiers, and Phase 4 aggregates and inserts hyperdocs. Each phase depends only on the outputs of prior phases. This sequential dependency chain is what makes the pipeline predictable and debuggable.

When a script lives in phase_0_prep/ but reads from Phase 3, it violates this contract. Anyone looking at the directory structure would reasonably assume that all scripts in phase_0_prep/ can run during Phase 0 — that is, before any LLM agents have been invoked. The four violating scripts cannot. They silently fail or produce empty results if run during actual Phase 0 processing, because the files they read from do not yet exist.

The interactive_batch_runner.py case is less severe. It lives in phase_1_extraction/ and reads Phase 2 files, but only to check whether they exist (for progress reporting). It does not process their contents. This is a pragmatic batch tool that needs visibility into later phases to report overall pipeline progress, and placing it in Phase 1 is defensible since its primary function is orchestrating Phase 1 extraction runs.

The Root Cause

The four misplaced scripts — completeness_scanner.py, schema_normalizer.py, normalize_agent_output.py, and schema_validator.py — were all written as utilities for working with agent output. They ended up in phase_0_prep/ because the schema normalization logic was originally developed alongside the Phase 0 enrichment code, and the completeness scanner was built to validate Phase 0's output before it evolved to check all phases. As the pipeline grew, these scripts accumulated dependencies on later phases without being relocated.

Recommendation: Move completeness_scanner.py, schema_normalizer.py, normalize_agent_output.py, and schema_validator.py from phase_0_prep/ to tools/. They are cross-phase utilities, not Phase 0 processing steps. This change is purely organizational — the code itself does not need to change, only the file location and any import paths that reference them. Update phase_0_prep/__init__.py if it exports these modules, and update any scripts that import from phase_0_prep.schema_normalizer to import from tools.schema_normalizer instead.
