Code Similarity Engine

9 ideas
  • Code matching must be completely unobstructed
    ArchitecturePriority
    Do NOT let genealogy pre-filter which files get compared. Compare everything against everything. The most valuable discoveries are files that don't look related by name but share significant code. Independence from genealogy is a design constraint.
  • Code similarity explains why individually-passing files break together
    VisionPriority
    6 files each pass their own tests. App breaks. Code similarity finds the interface drift — a module in one file that needs to align with modules in other files doesn't match. This is the core use case. Not archaeology. Live diagnostics.
  • Code similarity should run all the time
    Product
    Like a type checker — you don't run it once. An analysis step can easily turn into an ongoing real-time scan. But dead apps in standby mode need to be leveled up to real-time. They can get there, but there's a progression.
  • Name overlap without code overlap is its own signal
    Technical
    Files with matching names but no matching code means someone rewrote from scratch. Did they lose important logic? Also: files with <1% code match across all other files tells a lot — either truly independent or disconnected from the system it should be part of.
  • Taxonomy of similarity patterns — index them all
    Architecture
    As long as we can index all similarity pattern types, and use Opus to read them and get full context for each file, that will be enough to understand the patterns. The taxonomy is the foundation.
  • Archaeology mode for dead apps
    ProductMarketing
    Users with lots of dead apps want to bring them back to life. The archaeological scan tells them what they have. Framing matters — how you present "here's your dead code" determines whether users feel empowered or overwhelmed.
  • Similarity taxonomy: 8 pattern types
    Technical
    Function clone, structural mirror, partial extraction, template variant, evolution pair, import fingerprint, dead copy, logic overlap. Each pattern type has different implications for what to do about it.
  • Two modes: full scan + incremental
    Architecture
    Full scan: compare everything against everything (batch, run once). Incremental: re-compare changed files against known relationships (real-time, on every edit via PostToolUse hook). Mode 1 first, Mode 2 makes it a product.
  • Code similarity is independent of chat history
    ProductArchitecture
    This engine reads source code, not conversations. A user who never processes a single session can still use it. Standalone capability that integrates with but doesn't depend on the rest of the pipeline. Dependency flows: similarity → genealogy → visualizations.

The Leveling System

3 ideas
  • Apps in standby mode can't jump to real-time
    VisionPriority
    You can't just real-time past apps that are in standby mode. They can get there, but they have to be leveled up. There's a progression from dead → understood → enhanced → monitored.
  • Analysis naturally evolves into real-time scans
    Product
    An analysis step can easily turn into an ongoing real-time scan. The transition from batch to continuous is natural — but it requires the foundational analysis to exist first.
  • Codebase maturity levels
    Product
    Level 0: Dead app on disk. Level 1: Archaeological scan (similarity + genealogy). Level 2: Chat history processed (semantic primitives, idea graph). Level 3: Hyperdocs embedded. Level 4: Real-time monitoring. Each level builds on the previous.

Visualizations & Dashboard

8 ideas
  • Tableau-quality data visualizations, not styled text
    ProductPriority
    Stacked bar charts (messages per day/week/month). Interactive control panels. File genealogy trees. Idea evolution graphs. Every log detailed. All analyses visible. "Multi-color text on black background is not a visualization."
  • Control panel showing execution per file
    Product
    Every file as it's executed on every JSON/data file. All the analyses, implications, Opus recommendations. A paying customer should see every step of processing.
  • Manufacturing site tour — the system IS the experience
    VisionMarketing
    Design the pipeline visualization like a factory tour that gets visitors all the time. Transparent, impressive, reassuring. Show exactly how hyperdocs are made. Can be as cool or as boring as we want.
  • Production line → production surface (parallel agents)
    Vision
    When multiple agents work simultaneously, the production line becomes a production surface. Visualize in 3D with Three.js. Messages flowing through stations, branching to parallel agents, converging at synthesis.
  • 3 production-ready visualizations already exist
    Technical
    hyperdocs_dashboard.html (580KB, D3, real data), pipeline_visualization.html (352KB, D3, metrics), code_graph_studio.html (119KB, force graph). 5 others are broken/dead and should be removed or rebuilt.
  • The visualization IS the product
    Product
    95% of data collected, 97% analysis complete, 5% visible to humans, 2% usable by customers. The gap between "technically done" and "product" IS the visualization. Without it, only Claude can read the output.
  • Pipeline trace as visualization foundation
    TechnicalProduct
    Traced 3 messages through all 8 phases. A frustrated user message became a 0.99-confidence architectural constraint. Visualizing THIS transformation is the most compelling demo possible. Show the input, every station, and the output.
  • Chained visualization of the entire production line
    ProductVision
    Look at messages from different sessions and see the process for that text at each step. This shows whether steps need reordering, whether more layers are needed, and how to visualize the full chain.

Product & Marketing

9 ideas
"Context is not impossible. We can do it file by file... message by message."
— Stefan, core marketing message
  • The system has to be its own experience
    VisionPriority
    A system that deviates everyone will be shitty unless it's its own experience. The AI world is the most adaptive industry ever. The key has to fit the keyhole. Dialing back ambition to match what actually works.
  • Help users plan deployments based on time, money, data
    ProductMarketing
    Users who realize Opus is too expensive. Users who only want future work. Users with messy file systems. Help them plan based on any combination of constraints. The product adapts to the user's reality.
  • User preferences affect marketing — discover as you build
    Marketing
    As aspects of user preferences and constraints emerge during development, they inform the marketing message. The purest message is about context management. Everything else is a constraint to work around.
  • Multiple onboarding scenarios must be tested
    Product
    Messy file system. Future-only (skip historical). Budget user (Haiku only). 5 sessions vs. 500 sessions. Cancels mid-processing. Agent failures. Each scenario needs to work before launch.
  • Pricing tension: $19/month vs. compute costs
    Product
    $19/month doesn't cover Opus compute for even one session (~$15-30). Either the subscription covers platform + visualization (user brings API key), or pricing includes compute credits. This decision affects the entire marketing message.
  • The 10 commitments are the most valuable output
    Vision
    Evidence-based behavioral constraints derived from statistical analysis of real failures. Not rules — consequences. Each traces to a specific finding from 261 sessions. The pipeline's output isn't just documentation — it's behavioral infrastructure.
  • Semantic primitives are genuinely innovative
    Vision
    Everyone else doing "chat history analysis" does summarization. This system does structured extraction of the psychology of the development process. Seven dimensions per message. Action vectors, confidence, emotion, intent, friction, decisions, disclosure pointers. That's different.
  • Hyperdoc quality is real — B+ with A+ standouts
    Technical
    456/456 complete. Not boilerplate. P01 enforcer hyperdoc is genuine forensic analysis citing specific messages, tracing causality, documenting what's broken and why. Input: 1.4KB metadata. Output: 30KB+ archaeological report. Substantial value-add.
  • The product message: show users their own story
    MarketingVision
    "Here's what happened in your sessions. Here's where things went wrong. Here's what your AI forgot. Here's what you should read before your next session." Context is not impossible — and here's the proof, rendered in your own code.

Pipeline & Architecture

5 ideas
  • Pipeline state: 97% complete, 25 partial sessions
    Technical
    284 sessions, 6,135 JSONs, 456 hyperdocs, 341 enhanced files, 492 inputs, 13 genealogy families. 259 sessions fully complete (19 files each). 25 partial. Phase 5 ground truth needs re-run. 3 cron jobs keeping everything synced.
  • The transformation proof: frustration → constraint
    TechnicalVision
    Traced message 1108 ("ALWAYS WITH LLM!!!!!!!") through 8 phases. Entered as raw text with caps_ratio 0.89 and tier 4. Exited as GM-003: "LLM-first is mandatory" at 0.99 confidence. A frustrated outburst became a verifiable architectural constraint.
  • Code similarity feeds three downstream systems
    Architecture
    Similarity results feed into: (1) genealogy as a grounding signal, (2) ground truth verifier as a consistency check, (3) hyperdocs as per-file warnings about interface drift. One engine, three consumers.
  • Build code similarity BEFORE visualizations
    ArchitecturePriority
    The dependency chain: code similarity script → enriched genealogy → trustworthy visualizations. Building visualizations first means visualizing incomplete data. The script is the missing station on the production line.
  • Interface drift detection as product differentiator
    ProductVision
    "Your app doesn't work because these two files disagree about how data flows between them, and here's exactly where they diverge." Nobody else offers this. It's integration testing at the semantic level — not "do tests pass?" but "do these files agree?"
